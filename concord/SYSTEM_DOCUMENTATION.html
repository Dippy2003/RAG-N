Section 1

System Overview & Architecture

What is Concord?

Concord is a clinical record reconciliation system designed for the Sri Lankan healthcare ecosystem. When the same patient visits a clinic, a laboratory, and a pharmacy, each location creates its own record. These records often conflict — different spellings of a name, different blood types recorded, medications prescribed at one location that dangerously interact with medications prescribed at another.

Concord automatically detects these conflicts, uses AI to adjudicate them, flags drug interactions and allergy mismatches, and provides a chat interface that lets staff at any location query or update patient data in natural language.

Registration Agent (registration_agent.py)

Handles patient creation and demographic updates.

Registration Flow

Validate source prefix

CLN → CLINIC, LAB → LAB, PHM → PHARMACY. Auto-generate source_ref_id (e.g. CLN-003).

Insert into source_records

Write all demographic fields, medications array, allergies array.

Generate embedding

Call re_embed_record(source_ref_id) — encodes name+dob+nic+phone+address into a 384-dim vector using all-MiniLM-L6-v2.

Find or create cluster

Run cosine similarity search against all existing embeddings. If similarity ≥ 0.85, link to existing cluster. Otherwise, create a new cluster.

Update Flow

When demographic fields (name, dob, nic, phone, address) change, the record is re-embedded so the patient matching system stays accurate.

if updates.keys() & {"name", "dob", "nic", "phone", "address"}:
    re_embed_record(source_ref_id)   # updates embedding vector in Supabase

Reconciliation Agent & Prescription Agent

Reconciliation Agent (agent.py)

When two or more patient records are compared, the Reconciliation Agent finds conflicts and adjudicates them using AI.

Reconciliation Pipeline

Load all records

Fetches each source_ref_id's full record from source_records.

Detect conflicts

Compares fields pairwise. Flags: name mismatches, DOB differences, blood type conflicts, medication interactions, allergy vs prescription mismatches.

RAG lookup per conflict

For each conflict, calls retrieve_for_conflict(description, conflict_type) to get relevant clinical guidelines. Conflict type maps to the correct RAG category.

LLM Adjudication (Call 1)

Sends all conflicts + RAG guidelines to Gemini 2.0 Flash. Returns a resolution (accept_a / accept_b / escalate / flag_critical) with rationale and confidence for each conflict.

Safety Review (Call 2)

Sends proposed resolutions to LLM for a second safety pass. Flags anything needing human clinician review.

Write to DB + Notify ALL locations

Writes resolutions to adjudications table, escalations if needed. Creates a notification for every source_ref_id in the same cluster — not just the one being queried.

Conflict Types

allergy_mismatchallergy_mismatchPenicillin allergy + Amoxicillin prescribed
data_integritydata_integrityBlood type A+ (clinic) vs B+ (lab)

Search Implementation

def _search(filter_val: str) -> QueryResult:
    # Name search
    rows = sb.table("source_records").select("*")
             .ilike("name", f"%{filter_val}%").execute().data
    # Blood type
    rows += sb.table("source_records").select("*")
              .ilike("blood_type", f"%{filter_val}%").execute().data
    # Medications/allergies contains (Postgres @> operator)
    rows += sb.table("source_records").select("*")
              .contains("medications", [filter_val]).execute().data
Query + LLM formatting After the Query Agent returns raw rows, the API passes them to the LLM with retrieve_for_chat() RAG context to generate a human-readable response. The user sees a formatted summary, not raw JSON.

LLM Interface — Adjudication & Safety Review

Overview

The LLM Interface (llm_interface.py) wraps all LLM calls behind a single transport layer with automatic failover. Exactly 2 LLM calls are made per reconciliation run.

LLM Call 1 — Adjudication

Function: adjudicate(conflicts, guidelines_context)

Input: list of conflicts + RAG guidelines context

Output: AdjudicationResult — one resolution per conflict

LLM Call 2 — Safety Review

Function: review_actions(adjudication, original_conflicts)

Input: proposed resolutions + original conflicts

Output: EscalationReview — which resolutions need human eyes

RAG System — Knowledge Base & Retrieval

Architecture

The RAG (Retrieval-Augmented Generation) system provides clinical guidelines context to the AI agents at the moment of decision-making. Rather than baking clinical knowledge into the LLM prompt, it is retrieved dynamically based on the specific drugs, conflicts, or queries at hand.

User query / conflict
→
Embed (all-MiniLM-L6-v2)
→
pgvector cosine search
→
Category filter + rerank
→
LLM prompt

Per-Use-Case Thresholds & Categories

Use caseFunctionThresholdCategoriesRerank?
Prescription safetyretrieve_for_prescription0.30drug_interaction + allergy_mismatchYes
Conflict resolutionretrieve_for_conflict0.25Mapped from conflict_typeYes
General chatretrieve_for_chat0.35AllYes

Severity Re-ranking

Results are re-ranked by score = similarity × severity_weight, so critical guidelines surface above medium ones even at slightly lower cosine similarity:

SeverityWeightEffect
critical2.0×A 0.60 similarity becomes score 1.20
high1.5×A 0.60 similarity becomes score 0.90
medium1.0×Score = similarity
low0.7×A 0.60 similarity becomes score 0.42

retrieve_for_conflict

def retrieve_for_conflict(conflict_description, conflict_type="", top_k=4):
    category_map = {
        "drug_interaction": "drug_interaction",
        "allergy_mismatch": "allergy_mismatch",
        "data_integrity":   "data_integrity",
    }
    category = category_map.get(conflict_type, "")   # "" = all categories

    return retrieve_guidelines(conflict_description, top_k=4,
                               threshold=0.25, category=category)

Hybrid Search (keyword pre-filter)

The keyword parameter applies a post-vector-search tag filter. For example, when prescribing "metformin", the keyword hint "metformin" filters out guidelines that don't mention metformin in their tags or title — preventing false positives from other unrelated diabetes drugs.

if keyword:
    kw = keyword.lower()
    rows = [r for r in rows
            if kw in (r.get("tags") or "").lower() or kw in r.get("title", "").lower()]

match_guidelines Supabase RPC (SQL)

This PostgreSQL function must exist in your Supabase database:

CREATE OR REPLACE FUNCTION match_guidelines(
  query_embedding vector(384),
  match_threshold float,
  match_count int
)
RETURNS TABLE (
  id uuid,
  guideline_id text,
  category text,
  title text,
  content text,
  severity text,
  tags text,
  similarity float
)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT
    mg.id, mg.guideline_id, mg.category, mg.title, mg.content,
    mg.severity, mg.tags,
    1 - (mg.embedding <=> query_embedding) AS similarity
  FROM medical_guidelines mg
  WHERE 1 - (mg.embedding <=> query_embedding) > match_threshold
  ORDER BY similarity DESC
  LIMIT match_count;
END;
$$;

Cross-Location Conflict Notifications

When a reconciliation detects a conflict, the system notifies all locations in the same cluster — not just the location being queried. This ensures, for example, that both CLN-001 and PHM-001 (who are the same patient) both see the conflict notification in their respective location views.

# In api.py _fast_reconcile_text():
# 1. Get all source_ref_ids in the same cluster
cluster_id = record["cluster_id"]
siblings = sb.table("source_records")
             .select("source_ref_id")
             .eq("cluster_id", cluster_id).execute().data

# 2. Create a notification for each sibling
for sibling in siblings:
    sb.table("notifications").insert({
        "source_ref_id": sibling["source_ref_id"],   # CLN-001, LAB-001, PHM-001 all get one
        "message": conflict_summary,
        "type": "conflict",
        "read": False,
    }).execute()

Destructive Action Confirmation

Messages containing destructive keywords are intercepted before sending to the API. A red confirmation modal appears asking the user to confirm.

const DESTRUCTIVE_KEYWORDS = [
  "delete", "remove record", "drop", "wipe", "erase",
  "clear all", "remove patient", "purge"
]

function isDestructive(msg: string): boolean {
  return DESTRUCTIVE_KEYWORDS.some(kw => msg.toLowerCase().includes(kw))
}

// On send:
if (isDestructive(text) && !confirmed) {
  setConfirmPending(text)   // shows modal
  return
}

API Reference

Base URL: http://localhost:8080

Core Endpoints

POST /chat

Main chat endpoint. Routes to the correct agent based on intent classification.

Request:
{
  "message": "Prescribe metformin for CLN-001",
  "source_ref_id": "CLN-001"   // optional hint
}

Response (issued):
{
  "action": "issued",
  "source_ref_id": "CLN-001",
  "patient_name": "Perera Sunil",
  "drug": "metformin",
  "reason": "Metformin prescribed successfully.",
  "message": "Metformin 500mg prescribed for Perera Sunil."
}

Response (blocked):
{
  "action": "blocked",
  "drug": "ibuprofen",
  "reason": "Contraindicated in dengue fever — risk of haemorrhage.",
  "interactions_found": ["LK-002: Dengue Fever — NSAIDs Contraindicated"]
}

GET /notifications

Parameters:
  source: str   // "CLN" | "LAB" | "PHM" | "" (all)
  limit:  int   // default 20

Response:
[
  {
    "id": "uuid",
    "source_ref_id": "CLN-001",
    "message": "Conflict detected: blood_type mismatch with LAB-001",
    "type": "conflict",
    "read": false,
    "created_at": "2026-06-13T..."
  }
]

DELETE /notifications/{id}

Deletes a single notification by ID.
Response: { "deleted": true }

DELETE /notifications

Parameters:
  source: str   // delete all notifications for this prefix

Response: { "deleted": 5 }

Response: { "status": "ok", "version": "1.0" }
